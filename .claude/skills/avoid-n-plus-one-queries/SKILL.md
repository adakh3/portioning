# Avoid N+1 queries in list / aggregate views

**The bug, in one sentence:** a list endpoint runs a fixed base query, then fires
*one or more extra queries per row* during serialization — so latency grows with
the number of rows, even when nothing is being computed live (the data was
pre-generated by a cron/job).

We have hit this repeatedly across aggregate views. This skill is the standing
rule so we stop reintroducing it. Worked example: the AI follow-up review queue
(`bookings/views/followups.py`, `serializers/followups.py`,
`services/followup_scheduler.py`) — a `lead_days_stale` field fired 2 aggregate
queries **per draft**.

## The smell (learn to see it before profiling)

Any of these on a code path that returns many rows is a red flag:

1. **`SerializerMethodField` that touches the DB** — calls a helper, a
   `.filter()/.aggregate()/.count()`, `obj.related.all()`, or a model property
   that queries. Runs once **per row**.
2. **A model `@property` used in a serializer/template** that does a query
   (`self.items.count()`, `self.payments.aggregate(...)`, "latest child").
3. **Accessing `obj.fk.field`** without `select_related('fk')` — one query per
   row to load the related object.
4. **Accessing `obj.children.all()`** without `prefetch_related('children')`.
5. **A Python loop over a queryset that calls another query inside** (in a view,
   a stats aggregation, a `run_*` service).

If field values depend on *other rows or related sets*, assume N+1 until proven
otherwise.

## The fixes — pick by relationship shape

| You need… | Use | Cost |
|---|---|---|
| A **ForeignKey's** fields (`obj.lead.name`) | `select_related('lead', 'lead__assigned_to')` | joins into the 1 base query |
| A **reverse FK / M2M** set you iterate (`obj.dishes.all()`) | `prefetch_related('dishes')` | 1 extra query total, not per row |
| A **count / sum / max / latest** off a related set (`# of items`, `last message at`, `days since`) | `annotate(...)` with `Count` / `Sum` / `Max` / `Subquery(OuterRef(...))` | computed inside the base query |
| A count on a *filtered* subset | `annotate(x=Count('rel', filter=Q(...)))` | same |

**Never** compute a per-row aggregate inside a `SerializerMethodField` or a
property when the view returns a list. Annotate it onto the queryset instead.

### Annotate an aggregate as a Subquery (the pattern we used)

When the aggregate spans a relation (e.g. "latest reviewed follow-up for this
draft's lead"), use `Subquery(OuterRef(...))`:

```python
from django.db.models import DateTimeField, Max, OuterRef, Subquery

def _annotate_last_touch(qs):
    last_reviewed = (
        FollowUpDraft.objects
        .filter(lead=OuterRef('lead'), status__in=('sent', 'dismissed'))
        .values('lead').annotate(m=Max('reviewed_at')).values('m')
    )
    last_message = (
        WhatsAppMessage.objects.filter(lead=OuterRef('lead'))
        .values('lead').annotate(m=Max('created_at')).values('m')
    )
    return qs.annotate(
        _last_reviewed_at=Subquery(last_reviewed, output_field=DateTimeField()),
        _last_message_at=Subquery(last_message, output_field=DateTimeField()),
    )
```

Then the serializer reads the annotation off the row — **zero** per-row queries:

```python
def get_lead_days_stale(self, obj):
    if hasattr(obj, '_last_reviewed_at'):          # list path: annotated
        last = last_touch_from_parts(obj.lead.updated_at,
                                     obj._last_reviewed_at, obj._last_message_at)
    else:                                          # single-object path: fine to query
        last = lead_last_touch(obj.lead)
    return max((timezone.now() - last).days, 0)
```

### Keep one source of truth for the calculation

The logic for *which values matter* lived in one helper
(`last_touch_from_parts`) used by BOTH the annotated list path and the live
single-object path — so the fast path can't silently diverge from the slow one.
Do the same: split "fetch the parts" from "combine the parts."

### `hasattr` fallback for shared serializers

A serializer used by both list responses (annotated) and single-object
responses (approve/dismiss/create — not annotated) should branch on
`hasattr(obj, '_annotated_field')`: fast path when present, one-query fallback
when absent. One extra query for a single object is fine; per row in a list is not.

## Prove it with a test (required — it's a bug fix, and it regresses silently)

A query-count assertion is the only thing that stops this coming back, because
the code *looks* correct either way. Compare 1 row vs N rows — the counts must be
**equal**:

```python
from django.db import connection
from django.test.utils import CaptureQueriesContext

with CaptureQueriesContext(connection) as baseline:
    self.client.get(f'{self.BASE}?page_size=all')     # 1 row

for _ in range(4):
    add_row_with_related_data()                        # each exercises the aggregates

with CaptureQueriesContext(connection) as scaled:
    res = self.client.get(f'{self.BASE}?page_size=all')  # 5 rows

self.assertEqual(len(scaled), len(baseline),
                 "query count grew with row count — N+1 regression")
```

Prefer this equality form over `assertNumQueries(exact_n)` — it's robust to
unrelated middleware/auth queries and states the actual invariant (O(1) in rows).
Make each added row have the related data the aggregate reads, or the subquery
short-circuits and the test proves nothing.

## Checklist before shipping any list / aggregate view

- [ ] Every `obj.fk.…` access is covered by `select_related`.
- [ ] Every `obj.children.all()` iteration is covered by `prefetch_related`.
- [ ] Every count/sum/max/latest/"days since" is an `annotate`, not a
      per-row method/property.
- [ ] No `SerializerMethodField` or model property on the list path runs a query.
- [ ] Stats/breakdown loops over a queryset don't query inside the loop.
- [ ] A `CaptureQueriesContext` test asserts query count is constant vs row count.
- [ ] Frontend shows a **loading skeleton** while the list fetches — never render
      the empty-state ("nothing here") during load; it reads as a blank/broken page.

## Frontend note (the "blank page while loading" half)

Slow endpoints expose a second bug: a list page that does
`const { data = [] } = useSWR(...)` renders its **empty state** during the
fetch, which looks like a blank/broken page. Use the loading flag
(`const { data, isLoading } = useSWR(...)`) and render skeleton rows while
`isLoading`, then the empty state only when the loaded list is genuinely empty.
